# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

All frontend commands run from the `frontend/` directory:

- `cd frontend && npm install` — install dependencies
- `cd frontend && npm run dev` — start the Vite dev server
- `cd frontend && npm run build` — production build (this is what Vercel runs via `frontend/vercel.json`, output to `frontend/dist/`)
- `cd frontend && npm run preview` — serve the built `dist/` output locally
- `cd frontend && npm run test` — run the Vitest suite

Backend commands (Django/DRF, under `backend/`) run via `docker compose` from the repo root — see `docker-compose.yml`. **The compose service is named `web`** (not `backend`): run backend commands as `docker compose run --rm web python manage.py <cmd>`.

## Deployment & workflow

**GOLDEN RULE — never deploy to production without an explicit instruction from the user.**
All development is verified locally, then shipped to the **test/staging** environment. Promotion to **production** happens *only* when the user explicitly says so (e.g. "promote to production", "deploy to prod"). Do not promote to production proactively, "to finish the job", or because staging looks good — always stop and wait for the user's go-ahead. Approval to deploy to test is **not** approval to deploy to production.

### 1. Verify before merging (always)

Before opening/merging any PR, run and pass:
- `cd frontend && npx vitest run` — the full Vitest suite. **Run it from the `frontend/` directory.** If run from the repo root, Vitest globs stale `.worktrees/**` and `.claude/worktrees/**` copies (leftover git worktrees) that lack `node_modules` and produces a spurious mass failure ("hundreds of test files failed", `import` times in the thousands of seconds). That is an environment artifact, not a real failure — re-run scoped to `frontend/`.
- `cd frontend && npm run build` — the production build (this is what both Vercel and the prod `deploy.sh` run).
- `docker compose run --rm web python manage.py test` — the full Django suite (run the specific app(s) touched for speed, then the whole suite before a release).
- `docker compose run --rm web python manage.py makemigrations --check --dry-run` — catch model changes with no migration.

Add/adjust tests for any behavior change; `frontend/StaffDashboard.test.jsx` (~100 assertions) is treated as a behavioral contract — only change it when the change is intentional and note why.

### 2. Ship to test / staging (the normal flow)

Two remote branches gate integration: **`righteoushack`** (integration) and **`main`**. The per-change flow, established across sessions, is:

1. Work on a feature branch (e.g. `feature/launch-day-hardening`); commit with the repo's commit-message convention.
2. `git push origin <feature-branch>` → open a PR **into `righteoushack`**, merge it.
3. Open a PR **into `main`**, merge it.
4. **Test deploys automatically + manually:**
   - **Frontend**: Vercel watches `main` and rebuilds **test.theashantihub.com** on merge (Vercel "Root Directory" = `frontend`, per `frontend/vercel.json`). No action needed beyond the merge.
   - **Backend (staging)**: SSH in and run the staging deploy script:
     `ssh -i ~/.ssh/ashantihub_launch root@172.93.109.138` then `bash /opt/ashantihub/deploy-staging.sh`. It pulls `main`, migrates the **staging** DB (`/opt/ashantihub-staging/`, → **api-test.theashantihub.com**), and collects static. Skip it only when the change is frontend-only (no backend/migration diff).
5. Verify on test (endpoints return the expected status, migrations applied, the actual flow works in the browser). Then **stop** — production waits for the user.

### 3. Promote to production (only on the user's explicit say-so)

Production is **theashantihub.com** / **api.theashantihub.com**, served from **`/opt/ashantihub/`** on the same VPS (`172.93.109.138`). The `production` git branch tracks what's live. When — and only when — the user asks to promote:

1. **Back up the production DB first.** `deploy.sh` does **not** back up before migrating. Take a `pg_dump` (system Postgres, creds in `/opt/ashantihub/backend/.env`): write a gzipped dump to `/opt/ashantihub/backups/pre-promote-<timestamp>.sql.gz` and verify it (gzip -t, non-trivial `CREATE TABLE`/`COPY` counts) before proceeding.
2. **Fast-forward `production` → `main`:** `git push origin origin/main:refs/heads/production` (a clean fast-forward — no `--force`; production is always an ancestor of main under this flow).
3. **Run the prod deploy:** `ssh -i ~/.ssh/ashantihub_launch root@172.93.109.138` then `bash /opt/ashantihub/deploy.sh`. It resets to `origin/production`, `pip install`, **migrates the production DB**, collectstatic, **builds the frontend on the VPS** (`npm ci && npm run build`, rsync to `/var/www/ashantihub-frontend/` — production frontend is server-built, *not* Vercel), and restarts `ashantihub-gunicorn` + reloads nginx.
4. **Verify production** (frontend 200, a public API endpoint 200, a spot-check of the shipped feature). Flag any data migrations that delete/alter live data *before* running (e.g. a seed-clearing migration) so the user knows.

### Deployment gotchas

- **Server-side `.env` is not in git.** Settings like `GPS_REMOTE_VALIDATION` live only in `/opt/ashantihub/backend/.env` (and the staging equivalent). They persist across deploys (`deploy.sh` doesn't touch `.env`), but must be set per-environment by hand; back up the file before editing and restart `ashantihub-gunicorn` afterward.
- **Migrations run against live data on promote.** They applied cleanly on staging first (same ordered sequence), which is the safety net — always deploy to staging before production so migrations are proven against a real (separate) DB.
- The prod frontend is built on the VPS by `deploy.sh`; a stale browser cache can make a just-shipped UI change look "not deployed" — hard-refresh before diagnosing.

## Project structure

## Architecture

`App.jsx` is a monolith: dozens of components are defined as top-level `function` declarations before the single `export default function AshantiHub()` at the bottom of the file, which is the actual app root. Real client-side routing is provided by `react-router-dom`'s `BrowserRouter` (wrapped around `<App/>` in `main.jsx`, `docs/UI_MODERNIZATION_ROADMAP.md` Phase D) — but `AshantiHub` doesn't use `<Routes>`/`<Route>` matching machinery at all, since it's the sole component ever mounted at every URL. Instead it calls `useLocation()`/`useNavigate()` directly and derives `page` from the current pathname via a small `PATH_TO_PAGE` module-level map (`"/"`→`"home"`, `"/business"`→`"business"`, `"/events"`→`"events"`, `"/about"`→`"about"`, `"/contact"`→`"contact"`, `"/register"`→`"register"`), with a `setPage(id)` wrapper (the reverse `PAGE_TO_PATH` map + `navigate()`) standing in for the old bare `useState` setter — every existing `setPage("x")` call site (including `Navbar.jsx`, which is untouched by this and takes `setPage` as an opaque prop) keeps working unmodified. This fixed a real bug: `page` used to be local `useState` with zero URL sync, so visiting `/business` directly or hard-reloading while on it always bounced to home.

- `page` (derived from the URL, see above) switches sections within the main return via `page==="..."` conditionals. `/business/:id` and `/events/:id` are dynamic-segment paths matched via two `useMatch()` calls (`businessDetailMatch`/`eventDetailMatch` — not `<Routes>`/`<Route>` element matching, since `AshantiHub` stays the sole always-mounted component reading the router directly) rather than a `PATH_TO_PAGE` entry; when either matches, `page` resolves to `"business"`/`"events"` respectively so the surrounding tab chrome renders around the swapped-in detail page (see `selectedListingId`/`selectedEventId` below).
- Several boolean flags (`isAdmin`, `showBizDash`, `showPayments`, `showCredit`, `showAccount`) act like full-screen "routes": each one, if true, causes an **early return** of a full-page dashboard instead of rendering the normal marketplace UI — `isAdmin` → `StaffDashboard`, `showAccount` → `UserPanel` (the customer "My Account" page, see below), and `showBizDash`/`showPayments`/`showCredit` **all** → the single unified `BusinessCommandCenter` (`frontend/components/dashboard/`, `initialTab` chosen by which of the three flags is set), which superseded the three former separate `BusinessDashboard`/`PaymentDashboard`/`CreditDashboard` full-page dashboards (see the "Business Command Center" note below). The `BusinessDashboard` export kept in `App.jsx` is now a thin wrapper that renders `<BusinessCommandCenter initialTab="analytics" PaymentComponent={MoMoPayment}/>` (preserving the historical export/signature that `BusinessDashboard.test.jsx` imports). `isAdmin` is toggled by a hidden gesture (`handleLogoClick` — 5 clicks on the logo), or by visiting `/staff` directly (see below) — it stays local `useState`, not derived from the URL, since it can also flip true from that gesture or a post-login `onSuccess` while sitting on some other path (see the `/staff` bullet below for how it's kept in sync with the URL regardless). `showBizDash`/`showPayments`/`showCredit`/`showAccount` (`/business-dashboard`, `/payments`, `/credit`, `/my-account` — a `DASH_PATH_TO_FLAG`/`FLAG_TO_DASH_PATH` two-map pair, same convention as `PATH_TO_PAGE`/`PAGE_TO_PATH`) and `selectedListingId`/`selectedEventId` (`/business/:id`/`/events/:id`, via the `useMatch()`es described above) are all real, hard-reload-safe routes as of a follow-up `docs/UI_MODERNIZATION_ROADMAP.md` Phase D slice — each is now derived straight from `location.pathname` with its setter (`setShowBizDash`, `setSelectedListingId`, etc.) rewritten as a `navigate()`-calling closure rather than a raw `useState` setter, so every existing call site (`Navbar.jsx`, `BusinessRegistrationFlow.jsx`, `Card`'s `onOpen`, `ListingDetailPage`'s `onOpenListing`/`onBack`, `EventHeroCarousel`/`EventCard`'s `onOpen`, `EventDetailPage`'s `onBack`, and the inline CTA-band/Footer2 buttons in `App.jsx` itself) keeps calling them with their original signature (`setShowBizDash(true)`/`(false)`, `setSelectedListingId(id)`/`(null)`) unmodified. `showAccount`/`/my-account` was originally backed by a lightweight overlay (`AccountPanel`, since deleted) rather than a routed early return — it was brought into this same pattern once the destination grew into `UserPanel`, a full customer account page (see below). Two more early returns, `page==="staff-activate"` (`StaffActivatePage`) and `page==="reset-password"` (`ResetPasswordPage`) — staff onboarding + account-recovery work — follow this exact convention too, just keyed off `page` (via ordinary `PATH_TO_PAGE`/`PAGE_TO_PATH` entries, `/staff/activate` and `/reset-password`) rather than a dedicated boolean flag, since neither needs a `setShowX`-style toggle from elsewhere in the app — they're only ever reached by URL (an emailed link). See "Authentication is implemented end-to-end..." below for what each renders.
- Other flags (`showMessaging`, `showNotifs`, `showReferral`, `showMap`, `showCart`, cookie banner) render as overlays/modals on top of the normal page rather than replacing it. `showCart` (docs/BUSINESS_EVENTS_ROADMAP.md Phase 4) opens `CartDrawer`, gated to signed-in Customer accounts (`isCustomer`) same as its Navbar trigger.
- **`/staff` route:** `isAdmin` is not itself derived from the URL (it can also flip true via the 5-click gesture or a post-login `onSuccess` while sitting on some other path) — three effects keep it and the URL in sync instead: (1) a one-time (ref-guarded) deep-link check — once the session-restore fetch (`auth.isLoading`) settles, if the current path is `/staff` and the session isn't already staff, it opens the staff login modal (`setAuthModal("staff-login")`); (2) a `[location.pathname, auth.user, auth.isLoading]`-keyed effect that sets `isAdmin` to match whether the path is `/staff` *and* the session is staff — this is what makes hard-reloading on `/staff` while already logged in as staff work, and (with no manual `popstate` listener needed) what un-sets `isAdmin` when the browser's back/forward button navigates away from `/staff`; (3) an `[isAdmin]`-keyed effect that calls `navigate("/staff")`/`navigate("/")` to keep the URL following `isAdmin` for the gesture/post-login cases. Every page in the app is now URL-addressable — the five nav pages, `/register`, `/staff`, `/business-dashboard`, `/payments`, `/credit`, `/my-account`, `/staff/activate`, `/reset-password`, and the two dynamic-segment detail routes above; `frontend/vercel.json`'s catch-all SPA rewrite already serves all of them correctly with no config changes. Unmatched paths render `NotFoundPage` (a `show404` check against a small `KNOWN_PATHS` set — which now also includes the four dashboard paths alongside `/staff` — plus the two `useMatch()` results, since a `Set` of exact strings can't match a dynamic segment).

**`ModerationQueueTabs.jsx`** (`frontend/components/admin/`, staff moderation-queue restructuring) is the canonical Pending/Approved/Rejected sub-tab shell every moderated queue now sits on: `KYCQueuePanel`, `ListingsModerationPanel`, `HeroApprovalPanel`, `EventsModerationPanel`, `ReviewsModerationPanel`, `SubscriptionPlanApprovalPanel`, `DisputesPanel`, and `PromotionsPanel`. **Reuse it rather than hand-rolling a queue.** The panel owns its data (one react-query result per tab in a `queries={{pending, approved, rejected}}` object, all three fired on mount so the tab counts work) and passes `renderRow(item, state, index, items)` branching on `state`; the shell owns only the tab bar plus the active tab's loading/error/empty chrome. The editor computes its grant/revoke diff against the role's own permission set, which is why `StaffListSerializer` exposes `role_permissions` alongside the effective `permissions` — an effective permission *not* in the role set is an individual grant that must be preserved, which an effective-set-only view can't tell apart (getting this wrong would let `.set([])` silently clear a previously-granted permission). Enforcement of suspend/deactivate is two-layered: `StaffLoginSerializer` rejects at login, and `MultiAccountJWTAuthentication` refuses a still-valid token mid-session (staff-only — a live staff token reaches moderation/finance surfaces where an immediate cut matters, unlike the login-only block on customers/owners). `StaffListSerializer.get_status()` folds `deactivated`/`suspended` in ahead of the invite-derived states. A staffer can't suspend, deactivate, or edit the permissions of *their own* account (a `_guard_self_action` 400).

Authentication is implemented end-to-end against the real backend (`backend/accounts/`): `authModal`/`setAuthModal` (values `"login"`/`"signup"`/`"staff-login"`) render `AuthModal`, which uses `frontend/hooks/useAuth.js`'s `login`/`registerCustomer`/`registerBusinessOwner` against the customer/business-owner/staff login and registration endpoints. Its **signup** tab has the same Customer/Business Owner account-type toggle as its login tab (both driven by the same local `accountType` state) — picking Business Owner calls `registerBusinessOwner` instead of `registerCustomer`, and since that sets `registration_step: "business_info"` on the resulting user, `AshantiHub`'s existing `showRegistrationFlow` derivation (see below) automatically continues straight into `BusinessRegistrationFlow`'s KYC steps with no separate signup step required.

Every `AuthModal`'s login form (staff-login included) also has a **"Forgot password?"** link (staff onboarding + account-recovery work) switching to a third inline `mode="forgot"` — an email + Customer/Business Owner/Staff account-type toggle (hidden/fixed when `lockedAccountType` is set) calling `auth.requestPasswordReset(accountType, email)` (`POST /api/accounts/password-reset/request/`), which always resolves to the same generic "check your email" copy regardless of success/failure, matching the backend's own "never reveal whether an email matched an account" behavior. This only covers the *request* step inline in the modal; the *confirm* step (the emailed link itself, carrying `?token=&type=`) is its own standalone route, `/reset-password` → `ResetPasswordPage` (defined next to `AuthModal`, an early return in `AshantiHub` alongside `showAccount`/`showBizDash` rather than inside the normal `page==="..."` JSX switch, since it needs no Navbar/Footer chrome) — `POST /api/accounts/password-reset/confirm/`, success routes back to sign-in (`setAuthModal("login")` + `setPage("home")`) rather than logging the user in directly, since a password reset carries no session token. Staff invite activation follows the same shape: `/staff/activate` → `StaffActivatePage` (also an early return, reading `?token=`) calls the new `auth.activateStaff(token, password)` (`POST /api/accounts/staff/activate/`, mirrors `login()`'s own store-token-then-merge-`/me/` two-step exactly, since the activate endpoint only returns `{status, token}`, not a full login-response shape) and hands its result into `handleAuthSuccess` — the same `onSuccess` callback `AuthModal`'s login/signup already use (extracted out to a shared `AshantiHub`-level function for this reason), so a successful activation flips `isAdmin`/navigates to `/staff` exactly like a staff login would. Both new routes are added to `PATH_TO_PAGE`/`PAGE_TO_PATH` (and, by extension, `KNOWN_PATHS`, which spreads `PATH_TO_PAGE`'s keys) alongside the other five nav pages, but — like `showAccount`/`showBizDash` — are rendered via an early `if(page==="staff-activate")`/`if(page==="reset-password")` return rather than a `page==="..."` branch inside the main JSX tree.

## Styling

No CSS framework or CSS files are used — all styling is inline `style={{...}}` objects on JSX elements, using a shared color constant object `C` (Ghanaian/Ashanti-themed palette: gold, kente colors, Ghana flag colors, plus `pureBlack`/`white`) exported from `frontend/theme.js` and imported by both `App.jsx` and `frontend/components/*`. Reuse `C` rather than hardcoding new colors. The one other CSS mechanism in use is a component-local `<style>` tag holding raw `@keyframes`/`@media` rules (e.g. `LoadingScreen`'s loading-bar animation, `Hero`'s Ken-Burns drift, `Navbar`'s mobile-menu breakpoint) — still no CSS modules, framework, or external stylesheet.

**Exception, new work only:** per `docs/UI_MODERNIZATION_ROADMAP.md` (incremental TypeScript + Tailwind v4 + shadcn/ui adoption), `frontend/components/ui/*.tsx` components use real Tailwind utility classes + shadcn's CSS-variable design tokens instead of the `C`-palette inline-style convention. `frontend/index.css` scopes this deliberately — there's no global `body`/`html` Tailwind reset, so it doesn't fight the legacy inline-styled UI — every such component's root must be wrapped in `<div className="shadcn-scope">` (or have that class applied directly) to pick up the theme tokens at all. This exception is additive, not a migration: existing `C`-palette surfaces (`Card`, `Sidebar`, `ListingDetailPage`, etc.) are untouched and stay inline-styled until a later, separately-scoped pass.
