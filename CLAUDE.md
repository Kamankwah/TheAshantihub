# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

All frontend commands run from the `frontend/` directory:

- `cd frontend && npm install` — install dependencies
- `cd frontend && npm run dev` — start the Vite dev server
- `cd frontend && npm run build` — production build (this is what Vercel runs via `frontend/vercel.json`, output to `frontend/dist/`)
- `cd frontend && npm run preview` — serve the built `dist/` output locally
- `cd frontend && npm run test` — run the Vitest suite

Backend commands (Django/DRF, under `backend/`) run via `docker compose` from the repo root — see `docker-compose.yml`.

## Project structure

The repo is a monorepo with two self-contained top-level directories:

- `backend/` — Django/DRF/Postgres backend (see `backend/` for its own structure: `accounts/`, `listings/`, `core/`, `billing/`, `credit/` apps). `billing/` (`SubscriptionPlan`, `Subscription`, `Transaction`) and `credit/` (`CreditScore`) are minimal stub apps backing the `BusinessDashboard`/`PaymentDashboard`/`CreditDashboard` frontend dashboards with real (if naive/placeholder) data — not real Hubtel payment processing (`docs/HUBTEL_INTEGRATION.md`) or the real Phase-3 credit-scoring engine (`docs/PROJECT_SCOPE.md` §6), both of which remain future work.
- `frontend/` — the Vite + React app. Still a flat layout within `frontend/` — no `src/` subdirectory:
  - `frontend/index.html` — Vite entry HTML, loads `/main.jsx` as a module script and mounts to `#root`.
  - `frontend/main.jsx` — React root bootstrap (`ReactDOM.createRoot` + `<App />`, wrapped in `QueryClientProvider`).
  - `frontend/App.jsx` — the bulk of the application (~3,700+ lines). Most components, mock data, and business logic still live in this single file; `Navbar`/`Hero`/`HeroCarousel`/`Sidebar`/`ListingDetailPage` (see `frontend/components/` below) are the extractions out of it so far, per `docs/FRONTEND_MODERNIZATION.md` §2 — everything else stays in `App.jsx` until a future, separately-scoped extraction pass. `Card`, `MapView`, and the pure `groupCategoriesByKind` helper (Products/Services category-strip split, `docs/BUSINESS_EVENTS_ROADMAP.md` Phase 3) are still defined and `export`ed directly from `App.jsx` rather than moved to `frontend/components/` — they're unit-tested in place (`Card.test.jsx`, `MapView.test.jsx`, `groupCategoriesByKind.test.jsx` import from `./App.jsx`), matching this file's existing `BusinessDashboard`/`StaffDashboard`/`AuthModal` testing convention for components that haven't been extracted yet.
  - `frontend/components/` — small, standalone components extracted out of `App.jsx`. `App.jsx` still owns all state (`page`, `isAdmin`, `showBizDash`, search/filter state, etc.) and passes it down as props/callbacks; these components hold no app-level state of their own beyond purely local UI state (e.g. the mobile menu's open/closed flag).
    - `Navbar.jsx` — extracted from the old inline `Header` closure. Three-group layout (logo / centered nav links / right-aligned utility actions) with a hamburger/mobile menu (CSS `@media` breakpoint at 760px via a component-local `<style>` tag, same convention as `LoadingScreen`'s `@keyframes`) that collapses the utility action row — language toggle, notification bell, cart icon (Customer accounts only, `docs/BUSINESS_EVENTS_ROADMAP.md` Phase 4 — badge is the sum of cart item quantities, opens `CartDrawer` via a `setShowCart` prop), and the auth/profile menu — into a slide-down dropdown on narrow viewports. Messages/Saved/Payments/currency selector live elsewhere (floating `ChatLauncher`, the profile menu, the Business-tab currency selector) rather than in this action row.
    - `Hero.jsx` — extracted from the old inline hero JSX in the `page==="home"` branch. Replaced the single static `KUMASI_PHOTOS.manhyiaPalace` background with a multi-slide carousel over all `KUMASI_PHOTOS` entries (CSS opacity crossfade + a `heroKenBurns` `@keyframes` scale drift), with auto-advance (~5.5s) and manual prev/next/dot controls. Respects `prefers-reduced-motion`: auto-advance is paused and the Ken-Burns drift is skipped when the user's OS/browser has that preference set. The kente-gradient overlay, Ghana-flag-stripe bottom bar, search bar, and quick-action buttons are unchanged from before the carousel.
    - `Flag.jsx` — the small Ghana-flag SVG badge, extracted alongside Navbar/Hero (it's used by both `Navbar` and several places still in `App.jsx`) so it isn't duplicated across modules.
    - `HeroCarousel.jsx` — the Business tab's hero slider (`docs/BUSINESS_EVENTS_ROADMAP.md` Phase 3), sourced from `useActiveHero()` (approved, non-expired `HeroMediaSubmission` rows from Phase 2). Timer-based auto-rotate (~5.5s) with a CSS opacity crossfade plus prev/next/dot controls, adapted from `Hero.jsx`'s conventions but on an interval rather than scroll-pinned; pauses auto-advance (and swaps the crossfade to an instant cut) under `usePrefersReducedMotion`. Renders `null` entirely when there are no active hero submissions rather than a placeholder state.
    - `Sidebar.jsx` — reusable browse-filter sidebar (zone, price range, "verified businesses only" toggle, sort, clear filters), built generic enough for the Events tab (Phase 6) to reuse as-is. `AshantiHub` owns all the filter state and passes it down; Sidebar holds none itself. Desktop: static column in normal flow. Mobile (≤760px, same breakpoint as `Navbar.jsx`): becomes a fixed slide-in panel — toggled via an `open` prop/`ah-sidebar-open` class rather than inline `display`/`transform`, so the same non-`!important` CSS rule only takes effect once the mobile media query is active and never touches the desktop layout.
    - `ListingDetailPage.jsx` — the product/service detail page (PDP) for the Business tab grid. Not a top-level early return like `StaffDashboard`/`BusinessDashboard` — `AshantiHub` holds a `selectedListingId` state and swaps this in for the Sidebar+grid area only (inside the `page==="business"` block), so the hero carousel/WhatsApp banner/search bar/category tabs/CTA footer around it stay mounted. Primary data source is `useListing(id)` (promoted from favourites-drawer-only use) plus `useRelatedListings(id)` for the "Related" rail. Receives `CardComponent={Card}` as a prop (rather than importing `Card` from `App.jsx` directly) so `frontend/components/` doesn't grow a circular import back into `App.jsx`. "Add to Cart" (`docs/BUSINESS_EVENTS_ROADMAP.md` Phase 4) calls an `onAddToCart` prop — `AshantiHub`'s `handleAddToCart`, which auth-gates (must be a signed-in Customer), does the `POST /api/cart/items/`, and `refetch()`s the shared `useCart()` query so the Navbar badge/`CartDrawer` pick it up — following the same "AshantiHub owns the mutation, the component just calls the callback and keeps its own local adding/added/error UI state" convention as `onWhatsApp`/`onMessage`. Disabled when the listing has no price (nothing for the backend to charge) or while a request is in flight. There is no `specs`/`terms` field on the real `Listing` model, so no specs block is rendered (would otherwise be fabricated data).
    - `CartDrawer.jsx` — the customer cart (`docs/BUSINESS_EVENTS_ROADMAP.md` Phase 4), opened from `Navbar`'s cart icon via `AshantiHub`'s `showCart` flag. Mirrors `App.jsx`'s existing favourites drawer (`FavsDrawer`, still inline in `AshantiHub`) for visual/interaction consistency — same fixed-inset backdrop + top-right anchored panel — but extracted as its own file. Self-contained: calls `useCart()` itself (same convention as `ListingDetailPage` owning its own `useListing(id)`) rather than having `AshantiHub` thread cart data down as props; `AshantiHub` separately calls `useCart()` too, just for the Navbar badge count — same query key, so React Query shares the cache. Quantity +/- and remove are plain `apiPatch`/`apiDelete` calls followed by `refetch()`, matching this app's established mutation-handling convention (see `frontend/hooks/` below). Checkout is a three-step local `step` state (`"cart" → "confirm" → "confirmation"`): the confirm step's "Confirm & Pay" opens `PaymentComponent` — `App.jsx`'s `MoMoPayment` passed down as a prop (same "avoid an App.jsx ⇄ components/ circular import" convention as `ListingDetailPage`'s `CardComponent`) — whose `onSuccess` fires `POST /api/orders/checkout/` and whose `onClose` (the simulated payment's "Done" button) advances to the confirmation step once an order has actually been placed. Amounts are shown via the same GHS→display-currency conversion `Card`'s `displayPrice()` uses, sourced from `CURRENCIES` (now exported from `theme.js` alongside `C`, for exactly this cross-`components/`-boundary reason — it used to be a private constant in `App.jsx`).
  - `frontend/theme.js` — exports the shared `C` color palette object (previously defined inline at the top of `App.jsx`) and the `CURRENCIES` GHS-conversion rate table (ditto — moved here so `CartDrawer` could reuse `Card`'s exact conversion logic without a circular import). Single source of truth so `App.jsx` and `frontend/components/*` can import the same palette/rates without a circular `App.jsx` ⇄ `components/` dependency.
  - `frontend/apiClient.js` — shared `fetch` helper for calling the backend API. `apiFetch`/`apiPost`/`apiPatch`/`apiPostForm`/`apiPatchForm`/`apiDelete` cover GET/POST/PATCH/DELETE (JSON and multipart); the shared response handler treats a `204 No Content` (e.g. `DELETE /api/cart/items/{id}/`) as `null` rather than trying to parse an empty body as JSON.
  - `frontend/hooks/` — `@tanstack/react-query` data-fetching hooks: public marketplace data (`useCategories`, `useZones`, `useListings`, `useListing`, `useRelatedListings`, `useActiveHero`), staff-dashboard queues (`useKYCQueue`, `useModerationQueue`, `useHeroModerationQueue`, `useCustomers`, `useBusinessOwners`, `useStaffRoster`), business-owner self-service data (`useMyListings`, `useBusinessProfile`, `useSubscriptionPlans`, `useMySubscription`, `useMyTransactions`, `useMyCreditScore`, `useMyHeroSubmission`), and customer-scoped cart/order data (`useCart` — takes an optional `enabled` arg so callers can skip firing it for non-Customer accounts; `useOrders`). `useListings` accepts `kind` (`product`/`service`) and `verified` filter params alongside the existing ones, matching the backend's extended `/api/listings/` query params. Mutations are not wrapped in `useMutation` hooks — the established pattern is a plain `apiPost`/`apiPatch`/`apiDelete` call (from `apiClient.js`) inside the consuming component's event handler, in a `try/catch` that sets a local `actionError` state and calls the query's `refetch()` on success (see `KYCQueuePanel`, `BusinessDashboard`'s `saveEdit`, `CartDrawer`'s `updateQuantity`/`removeItem`). Checkout (`POST /api/orders/checkout/`) is likewise a one-shot `apiPost` inside `CartDrawer`'s handler rather than its own hook.
  - `frontend/mocks/`, `frontend/test/` — MSW request handlers and Vitest setup for the test suite.
  - `frontend/sw.js` — a service worker (not currently registered anywhere in `main.jsx`/`App.jsx`, so it's inert dead code unless registration is added).
  - `frontend/manifest.json` — PWA manifest, referenced from `index.html`.
  - `frontend/vercel.json` — Vercel build/routing config. **Vercel's dashboard "Root Directory" setting must be `frontend` for this to be picked up.**

## Architecture

`App.jsx` is a monolith: dozens of components are defined as top-level `function` declarations before the single `export default function AshantiHub()` at the bottom of the file, which is the actual app root. There is no React Router or any routing library — navigation is done entirely with local `useState` in `AshantiHub`, with one narrow, deliberate exception for `/staff` (see below).

- `page` (`"home" | "events" | "about"`) switches sections within the main return via `page==="..."` conditionals.
- Several boolean flags (`isAdmin`, `showBizDash`, `showPayments`, `showCredit`) act like full-screen "routes": each one, if true, causes an **early return** of a different full-page component (`StaffDashboard`, `BusinessDashboard`, `PaymentDashboard`, `CreditDashboard`) instead of rendering the normal marketplace UI. `isAdmin` is toggled by a hidden gesture (`handleLogoClick` — 5 clicks on the logo), or by visiting `/staff` directly (see below).
- Other flags (`showMessaging`, `showNotifs`, `showReferral`, `showMap`, `showCart`, cookie banner) render as overlays/modals on top of the normal page rather than replacing it. `showCart` (docs/BUSINESS_EVENTS_ROADMAP.md Phase 4) opens `CartDrawer`, gated to signed-in Customer accounts (`isCustomer`) same as its Navbar trigger.
- **`/staff` URL exception:** per `docs/PWA_STAFF_DASHBOARD.md` §4 Option B, `AshantiHub` uses raw `window.location.pathname`/`window.history.pushState` (no router library, no dependency added) so staff can land directly on `/staff` instead of only via the 5-click gesture. On mount, once the session-restore fetch (`auth.isLoading`) settles, if `pathname === "/staff"` it either sets `isAdmin` true (staff already logged in) or opens the staff login modal (`setAuthModal("staff-login")`). A second effect watches `[isAdmin]` and keeps the URL in sync — `pushState("/staff")` when it becomes true (gesture, direct visit, or post-login `onSuccess`), `pushState("/")` when `StaffDashboard`'s `onExit` sets it back to false. No other route in the app is URL-addressable; `frontend/vercel.json`'s catch-all SPA rewrite already serves `/staff` correctly with no config changes.

Most of `StaffDashboard` and the business-owner-facing dashboards (`BusinessDashboard`, `PaymentDashboard`, `CreditDashboard`) are wired to real backend data via the hooks above — `BusinessDashboard`'s Overview/Listings & Prices/Subscription tabs, `PaymentDashboard`, and `CreditDashboard` all read/write through `billing/`, `credit/`, `listings/`, and `accounts/` endpoints rather than local mock arrays. `CreditDashboard` shows only the signed-in business owner's own score (the backend has no aggregate/multi-business endpoint), so the old multi-business browsing UI (score cards grid, business-selector dropdown) was replaced with a single-score view. `PaymentDashboard` dropped its Invoices tab and "Revenue by Network"/"Active Subscribers" aggregate stats — there is no backend `Invoice` model and `Transaction` has no per-network breakdown, so these were removed rather than left as unbacked mock UI. `StaffDashboard` also has a "Hero Approval" tab (`HeroApprovalPanel`, gated by the `hero_media.approve` permission) approving/rejecting `HeroMediaSubmission` rows via `useHeroModerationQueue()` — clones `ListingsModerationPanel`'s shape exactly (`docs/BUSINESS_EVENTS_ROADMAP.md` Phase 2). `BusinessDashboard`'s Listings & Prices tab shows each listing's `photos` gallery (`OwnerListingSerializer` on `/api/listings/mine/` returns it) with a "Submit for Hero" action per photo (calls `POST /api/hero/submit/`), plus an inline status card sourced from `useMyHeroSubmission()` (`GET /api/hero/mine/` — the owner's outstanding submission if one exists, else their most recent one, else `{}`; absence of an `id` key, not response status, is the "nothing yet" signal, mirroring `useMySubscription`'s "no subscription yet" convention) with an "Extend" action once `status==="approved"` (reuses `MoMoPayment`'s simulated-payment flow, then `POST /api/hero/{id}/extend/`, then `refetch()`s the query — same mutation-handling pattern as the rest of this dashboard). Some data is still hardcoded in-file:

- `CATEGORIES` — the marketplace category list (hotels, food, tours, crafts, transport, etc.), each with an id/icon/label/color.
- `LISTINGS` — an object keyed by category id, each value an array of business listings (name, rating, price, location, phone, etc.) — this is the public/anonymous marketplace browse data, distinct from a signed-in business owner's own listings (`useMyListings`).
- `TRANSLATIONS` — i18n string tables, selected via the `lang` state (`T = TRANSLATIONS[lang]`).
- `mockEnquiries` (used by `BusinessDashboard`'s Enquiries tab) and `MOCK_CONVERSATIONS` (used by `MessagingCenter`) — real-time/AI messaging is a separate, larger Phase-2 initiative (`docs/PROJECT_SCOPE.md`) and is deliberately still mock.
- `LENDING_PARTNERS` — static lending-partner directory for `CreditDashboard`; no backend model exists for it (out of scope, frontend-only).
- `MOCK_CREDIT_BUSINESSES`, `SCORE_FACTORS` — leftover mock data still referenced only by `CreditCategoryView`, an inline marketing/explainer component that is defined but not currently rendered anywhere (dead code, pre-existing).
- `Analytics` — an in-memory event tracker (`Analytics.track`/`Analytics.getReport`); it does not send anything to a real endpoint, events just accumulate in a JS array for the session.

The Business tab (`page==="business"` in `AshantiHub`, ~line 3391 as of this writing — search for the `page==="business"` block and its "Business page — the marketplace browsing experience" comment) is a redesign per `docs/BUSINESS_EVENTS_ROADMAP.md` Phase 3: `HeroCarousel` sits above the existing WhatsApp banner; the category strip is split into Products/Services rows via `groupCategoriesByKind`; a `Sidebar` (zone, price range, "verified businesses only", sort, clear) replaces the old collapsible top-panel filter bar and sits alongside a denser multi-column `Card` grid (`useListings(filters)`'s existing infinite-query pattern, now with `kind`/`verified` added to `filters`); and a `selectedListingId` state in `AshantiHub` swaps the Sidebar+grid area for `ListingDetailPage` (the PDP) when a `Card`'s photo/name is clicked (`onOpen`), following the same "flag swaps in a component" convention as `isAdmin`/`showBizDash` but scoped inside the `page==="business"` block rather than a top-level early return, so the hero carousel/banner/search bar/category tabs/CTA footer stay mounted around the PDP. The grocery-concierge and transport-airport-info special-case panels, the Map/List toggle (`MapView`), and the closing CTA band are unchanged by this redesign, just relaid-out around the new Sidebar+grid structure.

Authentication is not actually implemented: `authModal`/`setAuthModal` and `user`/`setUser` state exist and are referenced throughout to gate features (WhatsApp contact, messaging, etc.), but there is no corresponding modal component that renders based on `authModal` — signup/login triggers currently no-op visually.

The "WhatsApp-first" contact pattern is central to the product: most business interactions (`handleWA`, `WABtn`) open a `wa.me` deep link with a prefilled message rather than an in-app contact form.

## Styling

No CSS framework or CSS files are used — all styling is inline `style={{...}}` objects on JSX elements, using a shared color constant object `C` (Ghanaian/Ashanti-themed palette: gold, kente colors, Ghana flag colors, plus `pureBlack`/`white`) exported from `frontend/theme.js` and imported by both `App.jsx` and `frontend/components/*`. Reuse `C` rather than hardcoding new colors. The one other CSS mechanism in use is a component-local `<style>` tag holding raw `@keyframes`/`@media` rules (e.g. `LoadingScreen`'s loading-bar animation, `Hero`'s Ken-Burns drift, `Navbar`'s mobile-menu breakpoint) — still no CSS modules, framework, or external stylesheet.
