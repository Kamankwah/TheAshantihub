# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

All frontend commands run from the `frontend/` directory:

- `cd frontend && npm install` — install dependencies
- `cd frontend && npm run dev` — start the Vite dev server
- `cd frontend && npm run build` — production build (this is what Vercel runs via `frontend/vercel.json`, output to `frontend/dist/`)
- `cd frontend && npm run preview` — serve the built `dist/` output locally
- `cd frontend && npm run test` — run the Vitest suite

Backend commands (Django/DRF, under `backend/`) run via `docker compose` from the repo root — see `docker-compose.yml`.

## Project structure

The repo is a monorepo with two self-contained top-level directories:

- `backend/` — Django/DRF/Postgres backend (see `backend/` for its own structure: `accounts/`, `listings/`, `core/`, `billing/`, `credit/` apps). `billing/` (`SubscriptionPlan`, `Subscription`, `Transaction`) and `credit/` (`CreditScore`) are minimal stub apps backing the `BusinessDashboard`/`PaymentDashboard`/`CreditDashboard` frontend dashboards with real (if naive/placeholder) data — not real Hubtel payment processing (`docs/HUBTEL_INTEGRATION.md`) or the real Phase-3 credit-scoring engine (`docs/PROJECT_SCOPE.md` §6), both of which remain future work.
- `frontend/` — the Vite + React app. Still a flat layout within `frontend/` — no `src/` subdirectory:
  - `frontend/index.html` — Vite entry HTML, loads `/main.jsx` as a module script and mounts to `#root`.
  - `frontend/main.jsx` — React root bootstrap (`ReactDOM.createRoot` + `<App />`, wrapped in `BrowserRouter` (`react-router-dom`, `docs/UI_MODERNIZATION_ROADMAP.md` Phase D) and `QueryClientProvider`).
  - `frontend/App.jsx` — the bulk of the application (~4,000+ lines). Most components, mock data, and business logic still live in this single file; `Navbar`/`Hero`/`HeroCarousel`/`Sidebar`/`ListingDetailPage`/`CartDrawer`/`EventHeroCarousel`/`EventCard`/`EventDetailPage`/`EventSubmissionPanel` (see `frontend/components/` below) are the extractions out of it so far, per `docs/FRONTEND_MODERNIZATION.md` §2 — everything else stays in `App.jsx` until a future, separately-scoped extraction pass. `Card`, `MapView`, and the pure `groupCategoriesByKind` helper (Products/Services category-strip split, `docs/BUSINESS_EVENTS_ROADMAP.md` Phase 3) are still defined and `export`ed directly from `App.jsx` rather than moved to `frontend/components/` — they're unit-tested in place (`Card.test.jsx`, `MapView.test.jsx`, `groupCategoriesByKind.test.jsx` import from `./App.jsx`), matching this file's existing `BusinessDashboard`/`StaffDashboard`/`AuthModal` testing convention for components that haven't been extracted yet.
  - `frontend/components/` — small, standalone components extracted out of `App.jsx`. `App.jsx` still owns all state (`page`, `isAdmin`, `showBizDash`, search/filter state, etc.) and passes it down as props/callbacks; these components hold no app-level state of their own beyond purely local UI state (e.g. the mobile menu's open/closed flag).

## Architecture

`App.jsx` is a monolith: dozens of components are defined as top-level `function` declarations before the single `export default function AshantiHub()` at the bottom of the file, which is the actual app root. Real client-side routing is provided by `react-router-dom`'s `BrowserRouter` (wrapped around `<App/>` in `main.jsx`, `docs/UI_MODERNIZATION_ROADMAP.md` Phase D) — but `AshantiHub` doesn't use `<Routes>`/`<Route>` matching machinery at all, since it's the sole component ever mounted at every URL. Instead it calls `useLocation()`/`useNavigate()` directly and derives `page` from the current pathname via a small `PATH_TO_PAGE` module-level map (`"/"`→`"home"`, `"/business"`→`"business"`, `"/events"`→`"events"`, `"/about"`→`"about"`, `"/contact"`→`"contact"`, `"/register"`→`"register"`), with a `setPage(id)` wrapper (the reverse `PAGE_TO_PATH` map + `navigate()`) standing in for the old bare `useState` setter — every existing `setPage("x")` call site (including `Navbar.jsx`, which is untouched by this and takes `setPage` as an opaque prop) keeps working unmodified. This fixed a real bug: `page` used to be local `useState` with zero URL sync, so visiting `/business` directly or hard-reloading while on it always bounced to home.

- `page` (derived from the URL, see above) switches sections within the main return via `page==="..."` conditionals. `/business/:id` and `/events/:id` are dynamic-segment paths matched via two `useMatch()` calls (`businessDetailMatch`/`eventDetailMatch` — not `<Routes>`/`<Route>` element matching, since `AshantiHub` stays the sole always-mounted component reading the router directly) rather than a `PATH_TO_PAGE` entry; when either matches, `page` resolves to `"business"`/`"events"` respectively so the surrounding tab chrome renders around the swapped-in detail page (see `selectedListingId`/`selectedEventId` below).
- Several boolean flags (`isAdmin`, `showBizDash`, `showPayments`, `showCredit`, `showAccount`) act like full-screen "routes": each one, if true, causes an **early return** of a different full-page component (`StaffDashboard`, `BusinessDashboard`, `PaymentDashboard`, `CreditDashboard`, `UserPanel`) instead of rendering the normal marketplace UI. `isAdmin` is toggled by a hidden gesture (`handleLogoClick` — 5 clicks on the logo), or by visiting `/staff` directly (see below) — it stays local `useState`, not derived from the URL, since it can also flip true from that gesture or a post-login `onSuccess` while sitting on some other path (see the `/staff` bullet below for how it's kept in sync with the URL regardless). `showBizDash`/`showPayments`/`showCredit`/`showAccount` (`/business-dashboard`, `/payments`, `/credit`, `/my-account` — a `DASH_PATH_TO_FLAG`/`FLAG_TO_DASH_PATH` two-map pair, same convention as `PATH_TO_PAGE`/`PAGE_TO_PATH`) and `selectedListingId`/`selectedEventId` (`/business/:id`/`/events/:id`, via the `useMatch()`es described above) are all real, hard-reload-safe routes as of a follow-up `docs/UI_MODERNIZATION_ROADMAP.md` Phase D slice — each is now derived straight from `location.pathname` with its setter (`setShowBizDash`, `setSelectedListingId`, etc.) rewritten as a `navigate()`-calling closure rather than a raw `useState` setter, so every existing call site (`Navbar.jsx`, `BusinessRegistrationFlow.jsx`, `Card`'s `onOpen`, `ListingDetailPage`'s `onOpenListing`/`onBack`, `EventHeroCarousel`/`EventCard`'s `onOpen`, `EventDetailPage`'s `onBack`, and the inline CTA-band/Footer2 buttons in `App.jsx` itself) keeps calling them with their original signature (`setShowBizDash(true)`/`(false)`, `setSelectedListingId(id)`/`(null)`) unmodified. `showAccount`/`/my-account` was originally backed by a lightweight overlay (`AccountPanel`, since deleted) rather than a routed early return — it was brought into this same pattern once the destination grew into `UserPanel`, a full customer account page (see below).
- Other flags (`showMessaging`, `showNotifs`, `showReferral`, `showMap`, `showCart`, cookie banner) render as overlays/modals on top of the normal page rather than replacing it. `showCart` (docs/BUSINESS_EVENTS_ROADMAP.md Phase 4) opens `CartDrawer`, gated to signed-in Customer accounts (`isCustomer`) same as its Navbar trigger.
- **`/staff` route:** `isAdmin` is not itself derived from the URL (it can also flip true via the 5-click gesture or a post-login `onSuccess` while sitting on some other path) — three effects keep it and the URL in sync instead: (1) a one-time (ref-guarded) deep-link check — once the session-restore fetch (`auth.isLoading`) settles, if the current path is `/staff` and the session isn't already staff, it opens the staff login modal (`setAuthModal("staff-login")`); (2) a `[location.pathname, auth.user, auth.isLoading]`-keyed effect that sets `isAdmin` to match whether the path is `/staff` *and* the session is staff — this is what makes hard-reloading on `/staff` while already logged in as staff work, and (with no manual `popstate` listener needed) what un-sets `isAdmin` when the browser's back/forward button navigates away from `/staff`; (3) an `[isAdmin]`-keyed effect that calls `navigate("/staff")`/`navigate("/")` to keep the URL following `isAdmin` for the gesture/post-login cases. Every page in the app is now URL-addressable — the five nav pages, `/register`, `/staff`, `/business-dashboard`, `/payments`, `/credit`, `/my-account`, and the two dynamic-segment detail routes above; `frontend/vercel.json`'s catch-all SPA rewrite already serves all of them correctly with no config changes. Unmatched paths render `NotFoundPage` (a `show404` check against a small `KNOWN_PATHS` set — which now also includes the four dashboard paths alongside `/staff` — plus the two `useMatch()` results, since a `Set` of exact strings can't match a dynamic segment).

Most of `StaffDashboard` and the business-owner-facing dashboards (`BusinessDashboard`, `PaymentDashboard`, `CreditDashboard`) are wired to real backend data via the hooks above — `BusinessDashboard`'s Overview/Listings & Prices/Subscription tabs, `PaymentDashboard`, and `CreditDashboard` all read/write through `billing/`, `credit/`, `listings/`, and `accounts/` endpoints rather than local mock arrays. `CreditDashboard` shows only the signed-in business owner's own score (the backend has no aggregate/multi-business endpoint), so the old multi-business browsing UI (score cards grid, business-selector dropdown) was replaced with a single-score view. `PaymentDashboard` dropped its Invoices tab and "Revenue by Network"/"Active Subscribers" aggregate stats — there is no backend `Invoice` model and `Transaction` has no per-network breakdown, so these were removed rather than left as unbacked mock UI. `StaffDashboard` also has a "Hero Approval" tab (`HeroApprovalPanel`, gated by the `hero_media.approve` permission) approving/rejecting `HeroMediaSubmission` rows via `useHeroModerationQueue()` — clones `ListingsModerationPanel`'s shape exactly (`docs/BUSINESS_EVENTS_ROADMAP.md` Phase 2). `BusinessDashboard`'s Listings & Prices tab shows each listing's `photos` gallery (`OwnerListingSerializer` on `/api/listings/mine/` returns it) with a "Submit for Hero" action per photo (calls `POST /api/hero/submit/`), plus an inline status card sourced from `useMyHeroSubmission()` (`GET /api/hero/mine/` — the owner's outstanding submission if one exists, else their most recent one, else `{}`; absence of an `id` key, not response status, is the "nothing yet" signal, mirroring `useMySubscription`'s "no subscription yet" convention) with an "Extend" action once `status==="approved"` (reuses `MoMoPayment`'s simulated-payment flow, then `POST /api/hero/{id}/extend/`, then `refetch()`s the query — same mutation-handling pattern as the rest of this dashboard). The same tab's edit form (any non-`published` listing) also edits two fields added to `Listing` by the reviews/Q&A backend slice (`backend/reviews/`, `backend/qa/` — new small apps, same granularity as `cart`/`orders`/`billing`/`credit`): a `service_duration` text input and a `specs` mini-editor (add/remove `{label, value}` rows), both included unconditionally in `saveEdit`'s `PATCH /api/listings/mine/{id}/` body — `OwnerListingSerializer.category` is a plain category id, not a nested `{kind, ...}` object, so there's no cheap client-side product/service check available here to branch `service_duration`'s visibility on; it's simplest to always render it (blank/unused for a product is harmless on a business owner's own private edit form). The same tab also has a "📣 Promote" action per published listing (`docs/BUSINESS_EVENTS_ROADMAP.md` Phase 5) — a small inline form (Featured/Boost `kind`, `days`, and a `keywords` field required only for Boost) that calls `POST /api/listings/{id}/promote/` directly; unlike Hero's submit/extend split, that single endpoint both validates and applies the promotion in one write and returns `amount_paid`, so the flow is apiPost-first, then `MoMoPayment` opens purely as the established simulated-pay UI step (`onSuccess` just `refetch()`s `useMyListings()` and shows the toast — there's no second apiPost). There's no "my promotions" list endpoint in this phase, so nothing is rendered once a promotion is purchased beyond the toast/refetch — nothing else to source a status card from, unlike `useMyHeroSubmission()`. `StaffDashboard`'s "Promotions" nav tab (`promotions.manage` permission) no longer renders the `ComingSoonPanel` placeholder now that promotion purchasing is a real, live, self-serve feature — it shows a small `PromotionsInfoPanel` explaining that business owners buy Featured/Boost promotions from their own dashboard, since there's no backend "list all promotions" endpoint in this phase to back a real admin management UI. `StaffDashboard` also has a "Site Settings" tab (`SiteSettingsPanel`, gated by the `site_settings.manage` permission, seeded onto `admin`/`super_admin`) editing the singleton `SiteSettings` row's footer/contact fields (`docs/UI_MODERNIZATION_ROADMAP.md` Phase B; 7 originally, now 9 — see below) — mirrors `CategoriesZonesPanel`'s CRUD-form shape (controlled inputs, plain `apiPatch` in a try/catch, local `actionError`, `.refetch()` on success) and `BusinessDashboard`'s `showToast`/`saved`-state convention for the "✓ Saved!" confirmation. Split into an outer `SiteSettingsPanel` (owns the `useSiteSettings()` query, renders loading/error states) and an inner `SiteSettingsForm` (only mounted once `settings.data` exists, so its form state is seeded once via a lazy `useState` initializer rather than a `useEffect` — avoids a race where an in-flight edit could get clobbered by the data arriving late). `SITE_SETTINGS_FIELDS` drives that form's field list generically; each entry can carry a `multiline: true` flag (used by the two policy-text fields the reviews/Q&A backend slice added — `warranty_returns_policy`/`service_dispute_policy`) to render as a `<textarea rows={4}>` instead of the default `<input>`, so `SiteSettingsForm` didn't need a second, duplicated form for those two fields. `StaffDashboard` also has a "Reviews" nav tab (`ReviewsModerationPanel`, gated by the `reviews.moderate` permission, seeded onto `admin`/`support`) — clones `ListingsModerationPanel`'s/`HeroApprovalPanel`'s shape but for hide/unhide rather than approve/reject, since `backend/reviews/` publishes a review immediately (gated on a real verified-purchase/attendee check at creation) rather than pre-moderating it: it lists every review regardless of status (a full reactive-moderation queue, not a pending-only one) with a "🚫 Hide" action (inline reason input, `POST /api/reviews/moderation/{id}/hide/`) for published reviews and "↩️ Unhide" (`POST .../unhide/`, no reason) for hidden ones. Its data hook, `useReviewsModerationQueue()` (`GET /api/reviews/moderation/`), is the one queue hook in this family that reads a paginated DRF envelope (`data?.results`, not `data||[]` like `useModerationQueue`/`useHeroModerationQueue`'s unpaginated array responses) — a real, callers-must-not-assume-unpaginated gotcha worth flagging since it looks identical to the other two panels at a glance. `StaffDashboard` also has a "Delivery Management" nav tab (`DeliveryManagementPanel`, gated by the `orders.manage_delivery` permission) listing every order via `useDeliveryQueue()` — same paginated-envelope shape/gotcha as the Reviews tab above, cloned closely from `ReviewsModerationPanel`'s shape (per-row action, local `actionError`, `refetch()` after each mutation). Only a `status==="paid"` order gets a `delivery_status` `<select>` (`processing`/`shipped`/`out_for_delivery`/`delivered`) — a pending/cancelled order has nothing to ship yet; changing it fires a plain `PATCH /api/orders/{id}/delivery-status/` in the same try/catch+refetch convention as the rest of this family, with the row's `<select>` disabled while that update is in flight.

`UserPanel` is the customer-facing "My Account" page, routed at `showAccount`/`/my-account` (see above) — an early-return full page rather than an overlay, replacing the old `AccountPanel` popover (deleted; it openly admitted "full profile editing isn't available yet" and had no avatar image, profile-editing, or order-history UI). Defined next to `StaffDashboard` in `App.jsx` (not `frontend/components/`) so it can reuse `MessagingCenter`/`MOCK_CONVERSATIONS`/`FavDrawerItem` (all module-top-level in this file) and `DASHBOARD_THEME`/`useTheme()` the same way `StaffDashboard` does — but, unlike `StaffDashboard`, skips the sidebar-collapse-toggle state entirely, since it only ever has 5 fixed, unfiltered nav items (`USER_NAV_ITEMS`): **Profile** (`ProfileTab` — name + avatar edit only, deliberately no email/phone, which are login identifiers with no verification/OTP flow yet; calls the new `auth.updateProfile({full_name, avatar})` — `useAuth.js`'s `updateProfile`, mirroring `submitBusinessInfo`'s FormData-building convention exactly — followed by the existing `refreshUser()`, same two-step convention `submitBusinessInfo` callers already use; a locally-picked `avatarFile` is previewed via a revoked-on-change `URL.createObjectURL`, falling back to `user.avatar` otherwise); **Orders & Delivery** (`OrdersDeliveryTab` — `useOrders()`, read-only, renders each order's items/total/date plus a `DeliveryStepper` only when `status==="paid"`, matching `DeliveryManagementPanel`'s delivery-status vocabulary but as a plain read-only step row, not a control); **Saved Businesses** (`SavedBusinessesTab` — the existing `favourites`/`toggleFav` state `FavsDrawer` already renders, reusing `FavDrawerItem` directly without the drawer's fixed-position overlay chrome); **Messages** (mounts the existing `MessagingCenter` as-is, its `onClose` flips the tab back to `"profile"` rather than closing anything, since `MessagingCenter` is itself a fixed full-viewport modal that stays that way here too); and **My Events** (`MyEventsTab` — `useMyEvents()`, the organizer's own submitted events only, each with a small local `EVENT_STATUS_META` status-pill map deliberately decoupled from `HERO_STATUS_META` even though the keys happen to match, plus a static "Attending history & tickets coming soon" note — no RSVP history or ticket-purchase UI, both explicitly out of scope). `user.avatar` (added to `AshantiHub`'s hand-picked `user` object, sourced from `/api/accounts/me/`'s `avatar` field for Customer accounts) is also what `Navbar.jsx`'s single avatar/profile-menu control (see below) renders as the real photo.

`Card` (`App.jsx`) and `ReviewsModal` (`App.jsx`) now render real review data instead of the dead scaffold they carried since before the reviews/Q&A backend existed — `Card`'s rating row (previously gated on a nonexistent `item.rating`) now gates on `item.review_count>0` and reads `item.avg_rating`/`item.review_count`, the top-level fields the real `Listing`/`Event` list/detail serializers now annotate onto every listing/event. `ReviewsModal` no longer reads the old hardcoded `MOCK_REVIEWS[item.id]` (deleted) — it owns its own `useListingReviews(item.id)` call for the review list/aggregate and, only when a `user` is signed in, a `useReviewEligibility({targetType:"listing",targetId:item.id})` call that renders one of three states in the "Write a Review" slot: the eligible star-picker+textarea form, "You've already reviewed this." (`already_reviewed`), or "You can review this after a completed purchase." (neither eligible nor already reviewed) — a signed-out visitor still sees the pre-existing "Sign in to leave a review" copy, unconditionally, without ever calling the eligibility endpoint. Submitting now does a real `apiPost("/api/reviews/", {target_type:"listing",target_id,rating,comment})` in the established try/catch+local-`actionError` convention, then `refetch()`s both the reviews list and the eligibility query so the aggregate/list and the "already reviewed" state update immediately without a page reload. `ReviewsModal` no longer takes an `onSubmit` prop — it's self-contained, matching `CartDrawer`/`ListingDetailPage`'s "owns its own mutation" convention. `EventCard.jsx` gained the equivalent net-new rating row (no dead scaffold existed there to activate) — a small local `starString()` helper rather than importing `Stars` from `App.jsx` (components/ never imports from App.jsx), same `review_count>0` hide-if-zero rule, sourced from the same `avg_rating`/`review_count` fields now present on the `GET /api/events/` teaser shape.

Some data is still hardcoded in-file:

- `CATEGORIES` — the marketplace category list (hotels, food, tours, crafts, transport, etc.), each with an id/icon/label/color.
- `LISTINGS` — an object keyed by category id, each value an array of business listings (name, rating, price, location, phone, etc.) — this is the public/anonymous marketplace browse data, distinct from a signed-in business owner's own listings (`useMyListings`).
- `TRANSLATIONS` — i18n string tables, selected via the `lang` state (`T = TRANSLATIONS[lang]`).
- `mockEnquiries` (used by `BusinessDashboard`'s Enquiries tab) and `MOCK_CONVERSATIONS` (used by `MessagingCenter`) — real-time/AI messaging is a separate, larger Phase-2 initiative (`docs/PROJECT_SCOPE.md`) and is deliberately still mock. Both are also staff-mediated per the "Businesses cannot be contacted directly" note below — neither represents a live customer<->business channel.
- `LENDING_PARTNERS` — static lending-partner directory for `CreditDashboard`; no backend model exists for it (out of scope, frontend-only).

## Styling

No CSS framework or CSS files are used — all styling is inline `style={{...}}` objects on JSX elements, using a shared color constant object `C` (Ghanaian/Ashanti-themed palette: gold, kente colors, Ghana flag colors, plus `pureBlack`/`white`) exported from `frontend/theme.js` and imported by both `App.jsx` and `frontend/components/*`. Reuse `C` rather than hardcoding new colors. The one other CSS mechanism in use is a component-local `<style>` tag holding raw `@keyframes`/`@media` rules (e.g. `LoadingScreen`'s loading-bar animation, `Hero`'s Ken-Burns drift, `Navbar`'s mobile-menu breakpoint) — still no CSS modules, framework, or external stylesheet.

**Exception, new work only:** per `docs/UI_MODERNIZATION_ROADMAP.md` (incremental TypeScript + Tailwind v4 + shadcn/ui adoption), `frontend/components/ui/*.tsx` components use real Tailwind utility classes + shadcn's CSS-variable design tokens instead of the `C`-palette inline-style convention. `frontend/index.css` scopes this deliberately — there's no global `body`/`html` Tailwind reset, so it doesn't fight the legacy inline-styled UI — every such component's root must be wrapped in `<div className="shadcn-scope">` (or have that class applied directly) to pick up the theme tokens at all. This exception is additive, not a migration: existing `C`-palette surfaces (`Card`, `Sidebar`, `ListingDetailPage`, etc.) are untouched and stay inline-styled until a later, separately-scoped pass.
