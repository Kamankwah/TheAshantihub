# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

All frontend commands run from the `frontend/` directory:

- `cd frontend && npm install` — install dependencies
- `cd frontend && npm run dev` — start the Vite dev server
- `cd frontend && npm run build` — production build (this is what Vercel runs via `frontend/vercel.json`, output to `frontend/dist/`)
- `cd frontend && npm run preview` — serve the built `dist/` output locally
- `cd frontend && npm run test` — run the Vitest suite

Backend commands (Django/DRF, under `backend/`) run via `docker compose` from the repo root — see `docker-compose.yml`.

## Project structure

The repo is a monorepo with two self-contained top-level directories:

- `backend/` — Django/DRF/Postgres backend (see `backend/` for its own structure: `accounts/`, `listings/`, `core/`, `billing/`, `credit/` apps). `billing/` (`SubscriptionPlan`, `Subscription`, `Transaction`) and `credit/` (`CreditScore`) are minimal stub apps backing the `BusinessDashboard`/`PaymentDashboard`/`CreditDashboard` frontend dashboards with real (if naive/placeholder) data — not real Hubtel payment processing (`docs/HUBTEL_INTEGRATION.md`) or the real Phase-3 credit-scoring engine (`docs/PROJECT_SCOPE.md` §6), both of which remain future work.
- `frontend/` — the Vite + React app. Still a flat layout within `frontend/` — no `src/` subdirectory:
  - `frontend/index.html` — Vite entry HTML, loads `/main.jsx` as a module script and mounts to `#root`.
  - `frontend/main.jsx` — React root bootstrap (`ReactDOM.createRoot` + `<App />`, wrapped in `QueryClientProvider`).
  - `frontend/App.jsx` — the bulk of the application (~4,000+ lines). Most components, mock data, and business logic still live in this single file; `Navbar`/`Hero`/`HeroCarousel`/`Sidebar`/`ListingDetailPage`/`CartDrawer`/`EventHeroCarousel`/`EventCard`/`EventDetailPage`/`EventSubmissionPanel` (see `frontend/components/` below) are the extractions out of it so far, per `docs/FRONTEND_MODERNIZATION.md` §2 — everything else stays in `App.jsx` until a future, separately-scoped extraction pass. `Card`, `MapView`, and the pure `groupCategoriesByKind` helper (Products/Services category-strip split, `docs/BUSINESS_EVENTS_ROADMAP.md` Phase 3) are still defined and `export`ed directly from `App.jsx` rather than moved to `frontend/components/` — they're unit-tested in place (`Card.test.jsx`, `MapView.test.jsx`, `groupCategoriesByKind.test.jsx` import from `./App.jsx`), matching this file's existing `BusinessDashboard`/`StaffDashboard`/`AuthModal` testing convention for components that haven't been extracted yet.
  - `frontend/components/` — small, standalone components extracted out of `App.jsx`. `App.jsx` still owns all state (`page`, `isAdmin`, `showBizDash`, search/filter state, etc.) and passes it down as props/callbacks; these components hold no app-level state of their own beyond purely local UI state (e.g. the mobile menu's open/closed flag).

So this owns its own unfiltered `useEvents({})` call (independent of the Events grid's filtered query, same "owns its own data source" convention as `HeroCarousel`/`useActiveHero`) and feeds `SlideCarousel` the first few results that have `hero_media` set. Also exports `formatEventDate`, reused by `EventHeroCarousel`/`EventDetailPage`. Clicking any tile calls `onOpen(id)` regardless of lock state — there's no separate in-grid unlock step; `EventDetailPage` itself renders the locked/code-entry state for a private event.

Data source is `useEvent(id)` (`GET /api/events/{id}/`), which may return either the full `EventDetailSerializer` shape or the safe teaser subset — told apart by `"address" in detail` (the teaser's field list never includes `address` at all, the full shape always does once the caller is allowed to see it), rather than a dedicated `is_locked` flag from the backend. The teaser case renders a "this event is private — enter the code" prompt that `POST`s to `/api/events/{id}/unlock/`; a successful unlock's response is kept in local `unlocked` state and rendered directly (no re-query). The full case renders description/address/event date/media gallery/a live `going_count` badge, plus a "Get Directions" link (`https://www.google.com/maps?q={lat},{lng}`) only when `lat`/`lng` are present, plus an RSVP section (`docs/BUSINESS_EVENTS_ROADMAP.md` Phase 7): an "I'm Going"/"Can't Go" toggle for signed-in customer accounts (`user` prop, `accountType==="customer"`), a "sign in to RSVP" prompt when signed out, and a disabled toggle plus explanatory note for business-owner accounts (RSVP is customer-only server-side — `POST /api/events/{id}/rsvp/` 403s a business owner). There is no backend endpoint to read "is the caller currently RSVP'd" — only `POST`/`DELETE /api/events/{id}/rsvp/` to set it, and the organizer-only `GET /rsvps/` attendee list, neither of which is a per-caller self-check — so `rsvpStatus` is tracked purely optimistically in local state, seeded from the mutation's response, and always starts at "not going" on page load/refresh; this is documented in-file as a known limitation rather than backed by an invented endpoint. A private event's RSVP reuses the `code` already entered to unlock the detail page above (no second prompt). A `400` from the RSVP `POST` (`{"detail": "This event is at capacity."}`) renders a capacity-full state, detected via `apiClient.js`'s thrown error now carrying `.status`/`.body` (see `frontend/apiClient.js` below) rather than any client-side capacity number, since `capacity` isn't part of `EventDetailSerializer`'s exposed fields. Below the form, `useMyEvents()` (`GET /api/events/mine/`) lists the caller's own events with status, `rejection_reason` when rejected, and a copy-able `access_code` for private ones. An approved-but-unpaid event gets a "Pay to publish" action — the amount shown is `visibility_days * 2` (GHS/day), duplicating `events/views.py`'s `EVENT_DAILY_RATE` purely for display since `POST /api/events/{id}/pay/` computes and charges it server-side in one step and doesn't return a pre-charge amount; snapshotted into local state before opening `PaymentComponent` (`App.jsx`'s `MoMoPayment`, passed down as a prop) — same "Phase 4 lesson learned" convention as `BusinessDashboard`'s `heroExtendDays*5`. `onSuccess` then fires the actual `POST /api/events/{id}/pay/` and `refetch()`s. Each event in the list also gets a "👥 Attendees" toggle (`docs/BUSINESS_EVENTS_ROADMAP.md` Phase 7) that expands a local `EventAttendeesPanel` subcomponent (defined in this same file, not a separate `frontend/components/` file — small and tightly coupled enough not to warrant one) — it calls `useEventAttendees(eventId, { enabled: true })` (`GET /api/events/{id}/rsvps/`, organizer/staff-only, DRF-paginated `{count, next, previous, results}`) only once that specific event's panel is expanded, so attendee lists aren't eagerly fetched for every event in "My Events" up front; lists each `going` attendee's name/phone/email/RSVP date, with a "no one has RSVP'd yet" empty state and no "load more" pagination wired up (organizer can still see the total via `data.count`).
  - `frontend/theme.js` — exports the shared `C` color palette object (previously defined inline at the top of `App.jsx`) and the `CURRENCIES` GHS-conversion rate table (ditto — moved here so `CartDrawer` could reuse `Card`'s exact conversion logic without a circular import). Single source of truth so `App.jsx` and `frontend/components/*` can import the same palette/rates without a circular `App.jsx` ⇄ `components/` dependency.
  - `frontend/apiClient.js` — shared `fetch` helper for calling the backend API. `apiFetch`/`apiPost`/`apiPatch`/`apiPostForm`/`apiPatchForm`/`apiDelete` cover GET/POST/PATCH/DELETE (JSON and multipart); the shared response handler treats a `204 No Content` (e.g. `DELETE /api/cart/items/{id}/`) as `null` rather than trying to parse an empty body as JSON. On a non-2xx response it throws an `Error` carrying `.status` (the HTTP status) and `.body` (the best-effort parsed JSON error body, `null` if there wasn't one) alongside the usual message string — added in Phase 7 so a caller that needs to distinguish *why* a request failed (e.g. `EventDetailPage`'s RSVP flow telling a `400` "at capacity" apart from any other error) can do so without every call site re-implementing response parsing.
  - `frontend/hooks/` — `@tanstack/react-query` data-fetching hooks: public marketplace data (`useCategories`, `useZones`, `useListings`, `useListing`, `useRelatedListings`, `useActiveHero`, `useSiteSettings` — `GET /api/core/site-settings/`, public/no-auth, always `200` with the 7-field footer/contact object, self-healing on the backend so it never 404s, per `docs/UI_MODERNIZATION_ROADMAP.md` Phase B), staff-dashboard queues (`useKYCQueue`, `useModerationQueue`, `useHeroModerationQueue`, `useCustomers`, `useBusinessOwners`, `useStaffRoster`), business-owner self-service data (`useMyListings`, `useBusinessProfile`, `useSubscriptionPlans`, `useMySubscription`, `useMyTransactions`, `useMyCreditScore`, `useMyHeroSubmission`), customer-scoped cart/order data (`useCart` — takes an optional `enabled` arg so callers can skip firing it for non-Customer accounts; `useOrders`), and events data (`docs/BUSINESS_EVENTS_ROADMAP.md` Phases 6-7): `useEvents(filters)` (`GET /api/events/`, an infinite query mirroring `useListings`' pattern but with only `category`/`zone`/`search` filters — no price/ordering/kind, since events have none of those), `useEvent(id)` (`GET /api/events/{id}/`, may resolve to either the full detail or the safe teaser shape — see `EventDetailPage.jsx` above for how that's told apart), `useMyEvents()` (`GET /api/events/mine/`, the caller's own events including `access_code`, always a plain array — no "nothing yet" single-object convention needed since an organizer isn't limited to one outstanding event), and `useEventAttendees(eventId, { enabled })` (`GET /api/events/{id}/rsvps/`, Phase 7, organizer/staff-only; `enabled` defaults to `false` so `EventSubmissionPanel`'s `EventAttendeesPanel` only fires it once a given event's attendee list is actually opened — there's deliberately no `useEventRSVP` hook, since `EventDetailPage`'s RSVP mutations are a plain `apiPost`/`apiDelete`-in-handler, same convention as everywhere else, and there's no "my status" GET to read into a query in the first place). `useListings` accepts `kind` (`product`/`service`) and `verified` filter params alongside the existing ones, matching the backend's extended `/api/listings/` query params. Mutations are not wrapped in `useMutation` hooks — the established pattern is a plain `apiPost`/`apiPatch`/`apiDelete` call (from `apiClient.js`) inside the consuming component's event handler, in a `try/catch` that sets a local `actionError` state and calls the query's `refetch()` on success (see `KYCQueuePanel`, `BusinessDashboard`'s `saveEdit`, `CartDrawer`'s `updateQuantity`/`removeItem`, `EventSubmissionPanel`'s submit/media-upload/pay handlers, `EventDetailPage`'s RSVP handlers). Checkout (`POST /api/orders/checkout/`) is likewise a one-shot `apiPost` inside `CartDrawer`'s handler rather than its own hook.
  - `frontend/mocks/`, `frontend/test/` — MSW request handlers and Vitest setup for the test suite.
  - `frontend/sw.js` — a service worker (not currently registered anywhere in `main.jsx`/`App.jsx`, so it's inert dead code unless registration is added).
  - `frontend/manifest.json` — PWA manifest, referenced from `index.html`.
  - `frontend/vercel.json` — Vercel build/routing config. **Vercel's dashboard "Root Directory" setting must be `frontend` for this to be picked up.**

## Architecture

`App.jsx` is a monolith: dozens of components are defined as top-level `function` declarations before the single `export default function AshantiHub()` at the bottom of the file, which is the actual app root. There is no React Router or any routing library — navigation is done entirely with local `useState` in `AshantiHub`, with one narrow, deliberate exception for `/staff` (see below).

- `page` (`"home" | "events" | "about"`) switches sections within the main return via `page==="..."` conditionals.
- Several boolean flags (`isAdmin`, `showBizDash`, `showPayments`, `showCredit`) act like full-screen "routes": each one, if true, causes an **early return** of a different full-page component (`StaffDashboard`, `BusinessDashboard`, `PaymentDashboard`, `CreditDashboard`) instead of rendering the normal marketplace UI. `isAdmin` is toggled by a hidden gesture (`handleLogoClick` — 5 clicks on the logo), or by visiting `/staff` directly (see below).
- Other flags (`showMessaging`, `showNotifs`, `showReferral`, `showMap`, `showCart`, cookie banner) render as overlays/modals on top of the normal page rather than replacing it. `showCart` (docs/BUSINESS_EVENTS_ROADMAP.md Phase 4) opens `CartDrawer`, gated to signed-in Customer accounts (`isCustomer`) same as its Navbar trigger.
- **`/staff` URL exception:** per `docs/PWA_STAFF_DASHBOARD.md` §4 Option B, `AshantiHub` uses raw `window.location.pathname`/`window.history.pushState` (no router library, no dependency added) so staff can land directly on `/staff` instead of only via the 5-click gesture. On mount, once the session-restore fetch (`auth.isLoading`) settles, if `pathname === "/staff"` it either sets `isAdmin` true (staff already logged in) or opens the staff login modal (`setAuthModal("staff-login")`). A second effect watches `[isAdmin]` and keeps the URL in sync — `pushState("/staff")` when it becomes true (gesture, direct visit, or post-login `onSuccess`), `pushState("/")` when `StaffDashboard`'s `onExit` sets it back to false. No other route in the app is URL-addressable; `frontend/vercel.json`'s catch-all SPA rewrite already serves `/staff` correctly with no config changes.

Most of `StaffDashboard` and the business-owner-facing dashboards (`BusinessDashboard`, `PaymentDashboard`, `CreditDashboard`) are wired to real backend data via the hooks above — `BusinessDashboard`'s Overview/Listings & Prices/Subscription tabs, `PaymentDashboard`, and `CreditDashboard` all read/write through `billing/`, `credit/`, `listings/`, and `accounts/` endpoints rather than local mock arrays. `CreditDashboard` shows only the signed-in business owner's own score (the backend has no aggregate/multi-business endpoint), so the old multi-business browsing UI (score cards grid, business-selector dropdown) was replaced with a single-score view. `PaymentDashboard` dropped its Invoices tab and "Revenue by Network"/"Active Subscribers" aggregate stats — there is no backend `Invoice` model and `Transaction` has no per-network breakdown, so these were removed rather than left as unbacked mock UI. `StaffDashboard` also has a "Hero Approval" tab (`HeroApprovalPanel`, gated by the `hero_media.approve` permission) approving/rejecting `HeroMediaSubmission` rows via `useHeroModerationQueue()` — clones `ListingsModerationPanel`'s shape exactly (`docs/BUSINESS_EVENTS_ROADMAP.md` Phase 2). `BusinessDashboard`'s Listings & Prices tab shows each listing's `photos` gallery (`OwnerListingSerializer` on `/api/listings/mine/` returns it) with a "Submit for Hero" action per photo (calls `POST /api/hero/submit/`), plus an inline status card sourced from `useMyHeroSubmission()` (`GET /api/hero/mine/` — the owner's outstanding submission if one exists, else their most recent one, else `{}`; absence of an `id` key, not response status, is the "nothing yet" signal, mirroring `useMySubscription`'s "no subscription yet" convention) with an "Extend" action once `status==="approved"` (reuses `MoMoPayment`'s simulated-payment flow, then `POST /api/hero/{id}/extend/`, then `refetch()`s the query — same mutation-handling pattern as the rest of this dashboard). The same tab also has a "📣 Promote" action per published listing (`docs/BUSINESS_EVENTS_ROADMAP.md` Phase 5) — a small inline form (Featured/Boost `kind`, `days`, and a `keywords` field required only for Boost) that calls `POST /api/listings/{id}/promote/` directly; unlike Hero's submit/extend split, that single endpoint both validates and applies the promotion in one write and returns `amount_paid`, so the flow is apiPost-first, then `MoMoPayment` opens purely as the established simulated-pay UI step (`onSuccess` just `refetch()`s `useMyListings()` and shows the toast — there's no second apiPost). There's no "my promotions" list endpoint in this phase, so nothing is rendered once a promotion is purchased beyond the toast/refetch — nothing else to source a status card from, unlike `useMyHeroSubmission()`. `StaffDashboard`'s "Promotions" nav tab (`promotions.manage` permission) no longer renders the `ComingSoonPanel` placeholder now that promotion purchasing is a real, live, self-serve feature — it shows a small `PromotionsInfoPanel` explaining that business owners buy Featured/Boost promotions from their own dashboard, since there's no backend "list all promotions" endpoint in this phase to back a real admin management UI. `StaffDashboard` also has a "Site Settings" tab (`SiteSettingsPanel`, gated by the `site_settings.manage` permission, seeded onto `admin`/`super_admin`) editing the singleton `SiteSettings` row's 7 footer/contact fields (`docs/UI_MODERNIZATION_ROADMAP.md` Phase B) — mirrors `CategoriesZonesPanel`'s CRUD-form shape (controlled inputs, plain `apiPatch` in a try/catch, local `actionError`, `.refetch()` on success) and `BusinessDashboard`'s `showToast`/`saved`-state convention for the "✓ Saved!" confirmation. Split into an outer `SiteSettingsPanel` (owns the `useSiteSettings()` query, renders loading/error states) and an inner `SiteSettingsForm` (only mounted once `settings.data` exists, so its form state is seeded once via a lazy `useState` initializer rather than a `useEffect` — avoids a race where an in-flight edit could get clobbered by the data arriving late). Some data is still hardcoded in-file:

- `CATEGORIES` — the marketplace category list (hotels, food, tours, crafts, transport, etc.), each with an id/icon/label/color.
- `LISTINGS` — an object keyed by category id, each value an array of business listings (name, rating, price, location, phone, etc.) — this is the public/anonymous marketplace browse data, distinct from a signed-in business owner's own listings (`useMyListings`).
- `TRANSLATIONS` — i18n string tables, selected via the `lang` state (`T = TRANSLATIONS[lang]`).
- `mockEnquiries` (used by `BusinessDashboard`'s Enquiries tab) and `MOCK_CONVERSATIONS` (used by `MessagingCenter`) — real-time/AI messaging is a separate, larger Phase-2 initiative (`docs/PROJECT_SCOPE.md`) and is deliberately still mock.
- `LENDING_PARTNERS` — static lending-partner directory for `CreditDashboard`; no backend model exists for it (out of scope, frontend-only).
- `MOCK_CREDIT_BUSINESSES`, `SCORE_FACTORS` — leftover mock data still referenced only by `CreditCategoryView`, an inline marketing/explainer component that is defined but not currently rendered anywhere (dead code, pre-existing).
- `Analytics` — an in-memory event tracker (`Analytics.track`/`Analytics.getReport`); it does not send anything to a real endpoint, events just accumulate in a JS array for the session.

The Business tab (`page==="business"` in `AshantiHub`, ~line 3391 as of this writing — search for the `page==="business"` block and its "Business page — the marketplace browsing experience" comment) is a redesign per `docs/BUSINESS_EVENTS_ROADMAP.md` Phase 3: `HeroCarousel` sits above the existing WhatsApp banner; the category strip is split into Products/Services rows via `groupCategoriesByKind`; a `Sidebar` (zone, price range, "verified businesses only", sort, clear) replaces the old collapsible top-panel filter bar and sits alongside a denser multi-column `Card` grid (`useListings(filters)`'s existing infinite-query pattern, now with `kind`/`verified` added to `filters`); and a `selectedListingId` state in `AshantiHub` swaps the Sidebar+grid area for `ListingDetailPage` (the PDP) when a `Card`'s photo/name is clicked (`onOpen`), following the same "flag swaps in a component" convention as `isAdmin`/`showBizDash` but scoped inside the `page==="business"` block rather than a top-level early return, so the hero carousel/banner/search bar/category tabs/CTA footer stay mounted around the PDP. The Events tab (`page==="events"` in `AshantiHub`, ~line 3812 as of this writing — search for the `page==="events"` block and its "Events page — a full Events tab rebuilt on..." comment) was previously 100% static (one fixed hero image, three hardcoded fake events) and is now built entirely on real `events` app data, per `docs/BUSINESS_EVENTS_ROADMAP.md` Phase 6: `EventHeroCarousel` sits at the top; a WhatsApp-style intro banner and a "Submit an Event" toggle (opening `EventSubmissionPanel`) sit below it; then a debounced search bar (own `eventSearchInput`/`eventFilters` state, independent of the Business tab's `filters`); a category strip filtered to `Category.kind==="event"` (`eventCategories`, computed the same client-side-filter way `groupCategoriesByKind` does for products/services); a `Sidebar` reused with `showPriceRange`/`showSort`/`showVerifiedToggle` all `false` (zone + clear only) alongside an `EventCard` grid (`useEvents(eventFilters)`'s infinite-query pattern, mirroring `useListings`); and a `selectedEventId` state that swaps the Sidebar+grid area for `EventDetailPage` (passed the same `user` object as everywhere else in `AshantiHub`, for Phase 7's RSVP role-gating), same "flag swaps in a component, scoped inside the page block" convention `selectedListingId`/`ListingDetailPage` established in the Business tab. `KUMASI_PHOTOS.akwasidae` is no longer referenced by this tab (still defined in `App.jsx`, now otherwise-unused) now that the hero is real event photos. RSVP/attendee-list data (Phase 7) lives entirely inside `EventDetailPage`/`EventSubmissionPanel` themselves (see `frontend/components/` above) — `AshantiHub` doesn't hold any RSVP-related state of its own.

Authentication is not actually implemented: `authModal`/`setAuthModal` and `user`/`setUser` state exist and are referenced throughout to gate features (WhatsApp contact, messaging, etc.), but there is no corresponding modal component that renders based on `authModal` — signup/login triggers currently no-op visually.

The "WhatsApp-first" contact pattern is central to the product: most business interactions (`handleWA`, `WABtn`) open a `wa.me` deep link with a prefilled message rather than an in-app contact form.

## Styling

No CSS framework or CSS files are used — all styling is inline `style={{...}}` objects on JSX elements, using a shared color constant object `C` (Ghanaian/Ashanti-themed palette: gold, kente colors, Ghana flag colors, plus `pureBlack`/`white`) exported from `frontend/theme.js` and imported by both `App.jsx` and `frontend/components/*`. Reuse `C` rather than hardcoding new colors. The one other CSS mechanism in use is a component-local `<style>` tag holding raw `@keyframes`/`@media` rules (e.g. `LoadingScreen`'s loading-bar animation, `Hero`'s Ken-Burns drift, `Navbar`'s mobile-menu breakpoint) — still no CSS modules, framework, or external stylesheet.
